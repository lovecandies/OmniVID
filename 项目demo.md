# 🚀 OmniVid：响应式多模态视频语义解析引擎

### 📌 项目定位

OmniVid 是一个专注于“长视频非结构化数据提纯”的企业级 AI 知识引擎。系统通过自研的 DAG 异步调度流水线与时间轴感知的 RAG（检索增强生成）架构，将长达数小时的音视频内容转化为高信息密度的思维导图、结构化博客、会议纪要，并支持千万级字数的跨视频精准问答，彻底解决传统长视频“难检索、耗时长、难沉淀”的痛点。

### 🌟 核心业务功能 \(User Features\)

#### 全平台视频一键提取与去重

- **多数据源接入：** 支持通过 URL 链接一键解析 B站、YouTube、抖音、小红书等主流视频平台内容，或直接上传本地 GB 级超大视频文件。

- **秒传与防抖：** 基于文件内容 MD5 指纹识别，实现全网级别的视频内容去重与秒传。别人解析过的视频，你可以直接在 50 毫秒内获取完整 AI 总结，拒绝重复等待。

#### 多模态 AI 智能总结引擎

- **一键生成多维资产：** 提供丰富的总结模板。一键将 3 小时的视频转化为：

    - **高颜值思维导图：** 自动提取层级大纲并渲染为可下载的思维导图。

    - **结构化笔记：** 包含核心观点、金句提取、长文博客撰写。

    - **商业/办公文档：** 自动生成标准的会议纪要格式或 PPT 宣讲大纲。

- **多大模型自由切换：** 内置模型路由网关，用户可根据需求随时在 DeepSeek\-V3、Qwen\-Max、GPT\-4o 等不同大模型之间无缝切换，兼顾极速响应与深度推理。

#### 交互式“时间轴感知”字幕流

- **精准时间锚点：** 依托后端的毫秒级音视频对齐算法，生成的视频字幕不仅仅是纯文本，更是带有时间戳的互动面板。

- **指哪打哪：** 用户在阅读生成的长篇总结或字幕时，只需点击任意一句话，视频播放器即可实现“秒级精准跳转”至对应的画面，彻底告别盲目拖拽进度条。

#### 个人知识库与跨视频对话 \(Contextual RAG\)

- **多视频聚合：** 支持将多达 50 个视频加入同一个“个人知识库”中进行统一管理。

- **AI 深度对话助理：** 针对生成的总结不满意，或者想要了解更深度的细节，可以通过对话框直接向 AI 追问。AI 能够跨越多个视频的上下文，结合“滑动窗口检索”，给出精准的解答，并附带视频来源与时间戳参考。

### 🛠️ 后端架构与工程亮点 \(Architecture Highlights\)

为了支撑上述丝滑的业务体验，本项目在后端架构上进行了深度的重构与创新：

- **DAG 异步事件驱动流水线：** 摒弃了传统的同步阻塞调用。利用 **RocketMQ** 构建了“下载\-\>提音\-\>ASR识别\-\>切片向量化\-\>AI总结”的轻量级有向无环图（DAG）流水线。每个节点独立重试机制，确保了 2 小时以上超长视频在弱网环境下的稳定解析，容错率极高。

- **极致降本的语义缓存 \(Semantic Cache\)：** 引入 **Redis 向量搜索**机制。当用户进行 AI 对话提问时，系统会计算问题向量并与历史提问进行余弦相似度比对。对于高相似度的问题，直接返回缓存结果，成功拦截超过 60% 的重复大模型 API 调用，极大降低了 Token 成本。

- **连续内存数据结构优化：**

- 在处理带有时间戳的海量字幕切片时，底层刻意选用 `ArrayList` 等连续内存结构，配合**滑动窗口算法 \(Sliding Window\)**，将长文本上下文的截取与检索时间复杂度优化至 $O(1)$，实现了在极低算力开销下保留时间语境。

- **无感状态推流 \(SSE\)：**

- 后端不再依赖低效的前端轮询，而是采用 **Server\-Sent Events \(SSE\)** 建立单向长连接。流水线上的每一步状态变更，都会由后端主动、实时地推送到前端面板，打造了极佳的用户心智体验。

# 🌟 项目名称：OmniVid \- 响应式多模态视频语义解析引擎

### 📝 项目简介

本项目是一个自主研发的企业级视频内容结构化理解与知识库平台。针对超长视频处理中“链路长、易中断、Token成本高昂”的核心痛点，**摈弃了传统的线性同步处理逻辑，自主设计了基于 DAG（有向无环图）的异步任务调度模型**。同时，为解决 AI 总结缺乏时间上下文的问题，创新性地研发了“带时间轴感知的上下文检索（Contextual RAG）”机制，在保障极低成本的同时，实现了视频帧级别的精准知识问答与多模态总结。

### 🛠️ 核心技术栈

- **基础框架：** Java 17, Spring Boot 3\.x, MyBatis\-Plus

- **AI 与检索层：** LangChain4j, PostgreSQL \(引入 pgvector 插件支持向量检索\)

- **中间件：** RocketMQ \(事件驱动\), Redis \(语义缓存与分布式锁\), MinIO

- **核心算法：** 滑动窗口算法 \(Sliding Window\)、多路召回、余弦相似度计算

### 💡 核心“自主创新”点（面试高光模块）

这里的包装重点在于“你发现了什么问题，然后你是怎么通过技术解决的”。

#### 创新点一：自研 DAG 异步流水线，解决长视频处理“易碎”痛点

- **传统做法的缺陷：** 一般开源项目就是写一个大方法，依次调用下载、提取音频、ASR、总结。如果 ASR 耗时过长导致超时，整个任务就失败了，必须从头再来。

- **你的创新落地：**

    - 将视频处理抽象为多个独立的节点（Node），利用 **RocketMQ** 构建了一套轻量级的 DAG（有向无环图）事件驱动流水线。

    - 上一个节点（如“提取音频”）完成后，发送带状态的 MQ 消息，触发下一个节点（“ASR识别”）。

    - 每个节点都有独立的重试机制和断点记录。如果网络波动导致大模型调用失败，系统只会重试当前节点，而不需要重新下载和提取音频。这极大提升了系统的容错率和健壮性。

#### 创新点二：带时间轴感知的上下文 RAG 机制（时间与空间复杂度的极致优化）

- **传统做法的缺陷：** 只是把文本切成一段段存入数据库，用户提问时检索出相关文本给 AI。这会导致信息碎片化（例如 AI 知道“这里提到了微服务”，但不知道是在视频第几分钟说的，也丢失了前后语境）。

- **你的创新落地：**

    - 在内存中，摒弃了松散的对象拼接，利用 `ArrayList` 物理内存连续的特性来存储带有 `[startTime, endTime]` 的结构化字幕块，确保后续读取操作的 $O\(1\)$ 时间复杂度。

    - 在检索阶段，运用**滑动窗口算法 \(Sliding Window\)**。当向量数据库命中某一句核心字幕时，算法会以该字幕为中心，向前和向后各扩展 3 个时间窗口的字幕。

    - 这种设计既没有增加无意义的全局 Token 消耗，又精准保留了语义上下文，使得大模型不仅能回答问题，还能附带返回“请参考视频 12分30秒\-13分15秒”的精准跳转链接。

#### 创新点三：基于 Redis 的多级语义缓存（Semantic Cache），实现极致降本

- **传统做法的缺陷：** 即使用户问了相同的问题（例如“总结一下核心内容”和“这视频讲了啥”），系统也会傻傻地去调用昂贵的第三方大模型 API。

- **你的创新落地：**

    - 自主设计了一套防抖与降本机制。当用户发起提问时，系统首先将用户的问题转化为“向量特征”。

    - 去 Redis 中检索历史提问向量，计算**余弦相似度**。如果发现相似度大于 0\.95（证明这两个问题意思基本一样），则直接从 Redis 返回历史生成的答案。

    - 这一创新在面对爆款视频的高并发提问时，成功拦截了超过 60% 的无效大模型 API 调用，极大地节约了资金成本并实现了毫秒级响应。

#### 创新点四：自适应大模型路由网关（Strategy 进阶版）

- **传统做法的缺陷：** 让用户手动切换模型，或者后台写死一个模型。

- **你的创新落地：**

    - 实现了一个智能路由网关。当后端接到任务时，会根据任务类型和文本长度（上下文大小）自动进行资源分配。

    - 如果是简单的“格式化提取”或“生成思维导图大纲”，系统通过策略模式自动路由给成本极低、速度极快的模型（如 DeepSeek\-V3）。

    - 如果是复杂的“跨多视频深度逻辑总结”，则路由给长文本推理能力更强的模型（如 Qwen\-Max 或 GPT\-4o）。这体现了你对系统架构灵活性和商业成本的双重把控。

1. **“你的滑动窗口算法在处理字幕时，具体是怎么实现的？”**

    - **应对：** 解释你先将字幕按照时间顺序存入 `ArrayList`。当向量检索返回目标字幕的索引 `i` 时，利用数组随机访问的特性，以 $O\(1\)$ 的时间复杂度快速截取子列表 `list.subList(Math.max(0, i - windowSize), Math.min(list.size(), i + windowSize + 1))`，从而高效获取上下文。

2. **“MQ 异步解耦后，前端怎么知道任务进行到哪一步了？”**

# 🛠️ 核心技术栈与后端工程实践 \(Backend Tech Stack \& Engineering\)

#### Java 核心与算法级性能优化 \(Core Java \& Data Structures\)

在面对长视频海量文本流和高并发状态管理时，深度结合 Java 集合框架底层特性与算法思维，突破性能瓶颈：

- **O\(1\) 策略路由引擎：** 彻底重构多平台解析与大模型调用的 `if-else` 臃肿代码。利用 `HashMap` 配合自定义注解和 Spring 的 Bean 生命周期，实现视频源与解析器的自动注册绑定。路由分发时以极低开销精准命中策略实现类。

- **双重二分查找与连续内存优化：** 针对“点击时间轴跳转”与“时间切片检索”功能，摈弃低效的遍历查询。底层采用连续内存分配的 `ArrayList` 严格按时间顺序存储字幕块（`SubtitleBlock`）。针对目标时间区间的搜索，手写**双重二分查找算法 \(Double Binary Search\)**，将长视频的字幕范围定位时间复杂度从 $O(N)$ 降维至极速的 $O(\log N)$。

- **高频滑动窗口与链表控制：** 针对 AI 连续对话的上下文记忆管理，考虑到高频的“淘汰历史对话（头删）”与“新增当前对话（尾增）”场景，摒弃动态数组的移位开销，选用 `LinkedList` 维护 Session 会话结构，确保头尾操作的极致性能。使用 `HashSet` 与 `TreeSet` 进行标签的高效去重与有序聚合。

#### 高并发架构与异步中间件体系 \(High Concurrency \& Middleware\)

针对视频处理特有的“长耗时阻塞”与“大文件 IO”痛点，构建了一套高可用、低延迟的后端基础底座：

- **RocketMQ 异步 DAG 流水线：** 引入消息队列将原本几十秒的同步阻塞请求，解耦为分布式的事件驱动架构。将视频全链路处理（抽音、ASR、切片、大模型）抽象为有向无环图 \(DAG\) 的消息流转。前端接口实现 50ms 内快速响应（Ack），系统吞吐量提升数倍。

- **Redisson 分布式锁与防抖拦截：** 针对高并发下的资源竞争与大文件重复提交，基于视频文件 MD5 指纹生成全局唯一 Key。利用 Redisson 配合 WatchDog 看门狗机制实现分布式锁，精准拦截并发环境下的重复上传，避免无效的存储 IO 与 AI 算力浪费。

- **Redis 高级应用与限流：** 利用 Redis Lua 脚本实现基于令牌桶 \(Token Bucket\) 的 API 限流策略，保障底层大模型 API 不被恶意刷量击穿。同时引入 Redis 构建“语义缓存库”，拦截高频重复提问。

#### 主流微服务生态与持久化方案 \(Spring Ecosystem \& Storage\)

- **Spring Boot 3\.x \& AOP 切面编程：** 基于现代化的 Spring Boot 构建后端微服务。熟练运用 Spring AOP 进行统一日志切面记录、权限鉴权拦截以及全局统一异常处理，保证了业务代码的极度纯粹与高内聚。

- **MyBatis\-Plus \& 关系型数据建模：** 采用 MySQL 存储核心业务实体关系（用户、知识库、视频元数据）。利用 MyBatis\-Plus 简化单表 CRUD 并在复杂业务逻辑中进行多表联查 SQL 调优，配合覆盖索引避免回表查询。

- **MinIO 分片续传与直传架构：** 搭建本地/云端 MinIO 对象存储集群。主导设计并实现了“Pre\-signed URL 预签名直传”与“大文件分片断点续传”机制，核心 Java 服务器仅处理轻量级的控制流与状态校验，将沉重的数据流完全卸载，极大地释放了服务器网络带宽。

#### AI 编排层与向量检索引擎 \(AI Orchestration \& RAG\)

- **LangChain4j 业务级编排：** 作为后端的 AI 中间层，深度集成 LangChain4j 框架，统一封装底层模型网关，无缝切换各大主流 LLM 厂商（如 DeepSeek、通义千问等）。

**项目名称：** OmniVid \- 响应式多模态视频语义解析引擎

**担任角色：** Java 后端研发工程师

**核心技术栈：** Java 17, Spring Boot 3\.x, MyBatis\-Plus, RocketMQ, Redis / Redisson, MinIO, PostgreSQL \(pgvector\), LangChain4j

**【项目背景与需求】**

本项目是一个基于大模型（LLM）的企业级长视频内容结构化提纯与跨视频知识库平台。传统视频解析面临**链路极长易中断（下载/抽音/ASR/总结）、超大文件并发冲突、大模型 Token 成本高昂，以及 AI 总结丢失时间轴语境**四大痛点。本项目旨在从底层重构多模态调度架构，提供低延迟、高容错、低成本的 AI 视频基建底座。

**【核心职责与技术亮点】**

- **DAG 异步事件驱动架构设计 \(解决链路脆弱与高延迟\)：**

- 针对长视频解析耗时长导致的 HTTP 线程阻塞问题，摒弃传统同步调用模型，引入 **RocketMQ** 构建了轻量级有向无环图 \(DAG\) 流水线。将“提取音频\-\>ASR识别\-\>切片\-\>大模型推理”拆解为独立事件驱动节点，配合节点重试机制，大幅提升弱网环境下的系统容错率，并将前端上传接口的同步响应时间**压缩至 50ms 以内**。

- **内存连续性优化与滑动窗口算法 \(解决时间轴上下文丢失\)：**

- 主导研发“时间轴感知”的 Contextual RAG 引擎。摒弃传统的碎片化文本拼接，底层利用 `ArrayList` 物理内存连续的特性存储带有时间戳的字幕结构体。配合自研的**滑动窗口算法 \(Sliding Window\)**，以 $O(1)$ 的时间复杂度快速截取核心问题前后的时间切片。在不增加大模型 Token 消耗的前提下，实现了前端毫秒级的“点击总结文本精准跳转视频对应画面”的高级交互。

- **多级语义缓存与限流降本机制 \(解决大模型 API 成本失控\)：**

- 为降低高频重复提问带来的算力浪费，创新性设计了基于 **Redis 向量检索**的语义缓存 \(Semantic Cache\) 层。通过计算用户提问的向量特征与历史记录的余弦相似度（阈值 \>0\.95 判定拦截），成功**拦截约 60% 的无效大模型 API 调用**。配合 Lua 脚本实现令牌桶限流，有效抵御恶意刷量请求。

- **海量文件防抖与分片直传架构 \(解决大文件并发 IO 瓶颈\)：**

- 重构大文件上传链路，将传统的服务器代理流转改造为 **MinIO 预签名 URL \(Pre\-signed URL\)** 前端直传模式，极大释放了 Java 核心服务器的网络带宽。配合视频文件 MD5 指纹，使用 **Redisson WatchDog** 实现分布式锁，精准拦截高并发场景下的重复解析请求，实现全网级别的“秒传”体验。

- **多态模型路由与无感状态推流 \(提升系统扩展性与体验\)：**

- 基于**策略模式**与 Spring 容器生命周期，设计了 $O(1)$ 复杂度的动态路由网关，消除了大模型切换与多平台解析中的大量 `if-else`。使用 **SSE \(Server\-Sent Events\)** 替代低效的 HTTP 轮询，由后端在 MQ 消费阶段主动向前端推送任务状态机流转进度，降低服务器连接数开销。
