## 高频考点地图

| 考点 | OmniVid 场景 | 回答入口 |
| --- | --- | --- |
| MySQL 唯一索引 | 视频 MD5 去重 | Redis 先挡并发，MySQL 唯一索引做最终兜底 |
| MySQL 事务 | 视频入库 + 任务创建 | 两个动作必须同成同败 |
| 事务隔离级别 | 并发上传、任务抢占 | 重点讲可重复读、幻读、间隙锁、唯一索引冲突 |
| 乐观锁 | 任务状态推进 | `version` 防止多个 worker 乱序更新 |
| 行锁 | 抢占待处理任务 | `SELECT ... FOR UPDATE` 适合强互斥 |
| B+Tree | 字幕时间轴查询 | `(video_id, start_ms)` 联合索引 |
| 最左前缀 | 字幕范围检索 | 先等值 video_id，再范围 start_ms |
| 覆盖索引 | 字幕轻量列表 | 只查时间范围时可减少回表 |

| 热点 Key | 爆款视频总结 | 本地缓存、分片 Key、限流保护 |

| GC Roots | 字幕对象泄漏 | 排查对象为何无法回收 |
| jstack | 任务卡死 | 看线程池线程是否阻塞在 IO/锁/模型调用 |
| jmap | 内存飙升 | 看大对象、字幕集合、缓存对象 |
| Filter | 跨框架请求处理 | CORS、编码、最外层过滤 |
| Interceptor | 登录态校验 | Controller 前解析用户上下文 |
| AOP | 日志和耗时统计 | 业务方法切面 |

| Bean 生命周期 | 策略路由注册 | 不同总结模板和模型策略自动注册 |
| MyBatis 批处理 | 字幕批量入库 | batch size 和索引维护成本 |

| 顺序消息 | DAG 节点顺序 | 同一 jobId 维度保证顺序 |
| 延迟消息 | 失败重试 | 节点失败后延迟重试 |
| SSE | 任务进度推送 | 单向通知比 WebSocket 更轻 |

| 记忆分层 | 多轮问答 | Redis 短期记忆，MySQL 长期历史 |

## 最容易被拷打的 10 个问题

### 1. 为什么 Redis 锁之外还要 MySQL 唯一索引？

回答入口：

> Redis 锁是性能优化，不是最终一致性边界。锁可能过期、误删或在服务重启时丢失，所以视频 MD5 去重必须由 MySQL 唯一索引兜底。

### 2. `@Transactional` 什么情况下会失效？

回答入口：

> OmniVid 里视频入库和任务创建必须在同一个事务内。我会把事务放在被外部调用的 public Service 方法上，避免同类自调用、private 方法、异常没触发回滚这些失效场景。

### 3. 线程池队列为什么不能无界？

回答入口：

> 长视频解析任务很重，如果用无界队列，突发上传会让任务无限堆积，最后变成内存问题。我会设置有界队列和明确拒绝策略，让系统可降级、可观测。

### 4. 如何保证任务节点重复执行不出错？

回答入口：

> 每个 DAG 节点都以 MySQL 状态机为准，更新时校验当前状态和 version。即使重复触发 ASR 节点，也只能从合法状态推进，不能覆盖已完成结果。

### 5. 字幕查询为什么能快？

回答入口：

> 字幕是按视频内时间轴访问的，所以用 `(video_id, start_ms)`。查询先等值命中视频，再按时间范围扫描，符合 B+Tree 最左前缀。

### 6. SSE 和 WebSocket 怎么选？

回答入口：

> 任务进度是服务端单向推给前端，SSE 足够轻。WebSocket 适合强双向通信，比如协同编辑或实时游戏。

### 7. 大文件上传怎么避免 OOM？

回答入口：

> 不把整个视频读成 `byte[]`，而是流式写文件或对象存储。JVM 只保留少量缓冲和元数据。

### 8. 缓存和数据库如何保持一致？

回答入口：

> 总结资产用 Cache Aside。读时先缓存，写时先更新 MySQL 再删除 Redis。对强一致要求不高的展示型数据，接受短暂最终一致。

### 9. Agent 怎么防幻觉？

回答入口：

> 回答必须基于检索到的字幕片段，并返回视频名和时间戳。如果检索不到证据，就拒答或提示证据不足。

### 10. MVP 没上 RocketMQ 怎么讲？

回答入口：

> 我第一版先用本地 DAG 把状态机、幂等和重试边界跑稳。代码层预留 dispatcher，后续替换 RocketMQ 不改变核心业务状态模型。

## 简短自我介绍版

> OmniVid 是我围绕 Java 后端和 AI Agent 求职设计的长视频知识解析项目。它把视频上传、异步解析、字幕检索、总结生成和知识库问答串成一条真实链路。后端重点不是简单 CRUD，而是用 MySQL 保证任务状态和去重一致性，用 Redis 做分布式锁、进度缓存、限流和语义缓存，用线程池 DAG 承接长耗时解析任务，最后用 Agent 工具调用实现带时间戳引用的视频问答。
