# 03. Java 后端与 AI Agent 面试 Playbook

## 回答总模板

所有技术点都按这个结构回答：

```text
业务痛点 -> 技术方案 -> 八股关键词 -> 可验证结果
```

不要先背概念。先讲 OmniVid 的业务场景，再自然展开八股。

## Java 并发钩子

### 本地轻量 DAG 任务队列

业务痛点：

- 视频解析链路长，HTTP 同步等待会超时。

技术方案：

- 上传接口只创建任务并快速返回。
- 后端用线程池异步执行 DAG 节点。
- 每个节点更新 MySQL 状态机和 Redis 进度。

八股关键词：

- ThreadPoolExecutor。
- 核心线程数。
- 最大线程数。
- 阻塞队列。
- 拒绝策略。
- CompletableFuture。
- 异步异常处理。

面试回答：

> 我没有让上传接口同步跑完整解析链路，而是把它拆成本地 DAG 任务。接口只负责落库和投递任务，真正的 ASR、切片、总结在线程池里执行。这里我重点关注线程池队列不能无界，否则长视频任务堆积会 OOM；拒绝策略也不能静默丢任务，需要把任务标记为 WAITING 或 FAILED。

可验证结果：

- 上传接口快速返回。
- 后台日志显示 DAG 节点推进。
- 任务失败能在页面看到失败原因。

### 并发上传

业务痛点：

- 多个请求同时上传同一个视频。

技术方案：

- Redis 锁拦截第一层并发。
- MySQL 唯一索引兜底。
- Java 层避免使用本地锁作为分布式互斥。

八股关键词：

- synchronized。
- ReentrantLock。
- CAS。
- AQS。
- 分布式锁。
- happens-before。

面试回答：

> 单机锁只能保护当前 JVM，部署多实例后无效。所以我把本地锁的使用范围限制在线程内资源保护，跨请求、跨实例的视频去重交给 Redis 分布式锁和 MySQL 唯一索引。

## JVM 钩子

### 大文件上传避免 OOM

业务痛点：

- 视频文件可能很大，不能一次读进内存。

技术方案：

- 使用流式上传。
- 后端只处理控制流和元数据。
- 文件落本地或对象存储。

八股关键词：

- 堆内存。
- 直接内存。
- GC Roots。
- Young GC / Full GC。
- OOM 排查。
- jmap。
- jstack。

面试回答：

> 视频上传不能用 `byte[]` 一次性接收，否则几个并发大文件就可能把堆打满。我会用流式处理，把文件写到对象存储或磁盘，MySQL 只保存路径和元数据。如果出现 OOM，会先看堆 dump，确认是文件缓冲、字幕对象还是缓存对象没有释放。

### 字幕切片内存

业务痛点：

- 长视频 ASR 后可能产生大量字幕片段。

技术方案：

- 分批写入 MySQL。
- 控制内存中字幕窗口大小。
- Agent 检索时只加载命中片段附近窗口。

八股关键词：

- 对象生命周期。
- Minor GC。
- 老年代晋升。
- 批处理。
- 内存泄漏。

## Spring 钩子

### Filter / Interceptor / AOP

业务痛点：

- 登录鉴权、接口日志、业务埋点分别处在不同层。

技术方案：

- Filter 处理最外层请求。
- Interceptor 处理 Spring MVC 层的鉴权和用户上下文。
- AOP 处理业务方法日志、耗时统计和权限切面。

八股关键词：

- Servlet Filter。
- HandlerInterceptor。
- Spring AOP。
- 动态代理。
- JDK 代理。
- CGLIB。

面试回答：

> OmniVid 里登录态解析更适合放在 Interceptor，因为它贴近 Controller 路由；统一耗时日志和异常监控放 AOP，因为它关注业务方法；如果是跨框架的字符编码、CORS 这类更底层逻辑，才更适合 Filter。

### Transactional 失效

业务痛点：

- 视频入库和任务创建必须原子成功。

技术方案：

- Service 公有方法加事务。
- 避免同类内部方法自调用导致事务失效。
- 明确异常回滚规则。

八股关键词：

- 事务传播。
- rollbackFor。
- 自调用失效。
- private 方法失效。
- 代理对象。

面试回答：

> 我把视频入库和 processing_job 创建放在同一个 Service 事务里。如果任务创建失败，视频记录也不能半成功。这里要注意 `@Transactional` 是基于代理的，同类内部调用可能不生效，所以事务边界要放在外部调用的 public 方法上。

## MyBatis 钩子

### 字幕批量入库

业务痛点：

- ASR 一次会产生大量字幕片段，逐条 insert 效率低。

技术方案：

- 批量 insert。
- 控制 batch size。
- 异常时记录失败任务并支持重试。

八股关键词：

- 批处理。
- JDBC batch。
- SQL 执行效率。
- 索引维护成本。

### N+1 查询

业务痛点：

- 视频详情页要展示视频、任务、总结、字幕概览。

技术方案：

- 明确聚合查询。
- 分页查询字幕。
- 避免循环查子表。

八股关键词：

- N+1。
- 懒加载。
- 分页插件。
- 动态 SQL。
- SQL 注入防护。

## MQ 与异步架构钩子

### 本地 DAG 到 RocketMQ 的演进

业务痛点：

- MVP 可以单机跑，但生产环境需要多实例消费和更强可靠性。

技术方案：

- 第一阶段用本地线程池 DAG。
- 抽象 `JobDispatcher` 接口。
- 二阶段替换为 RocketMQ producer/consumer。

八股关键词：

- 消息可靠性。
- 重复消费。
- 幂等消费。
- 顺序消息。
- 死信队列。
- 延迟消息。

面试回答：

> 我第一版没有直接上 MQ，是为了先把业务状态机和幂等边界打稳。但代码层会把任务投递抽象成 dispatcher，后续切 RocketMQ 时，核心变化只是投递和消费方式。消费端仍然依赖 MySQL 状态机做幂等，所以重复消费不会破坏任务状态。

## 网络与操作系统钩子

### SSE 实时进度

业务痛点：

- 前端需要实时看到解析进度，不想频繁轮询。

技术方案：

- 使用 SSE 单向推送任务状态。
- 前端断线后根据 jobId 重连。
- Redis 保存最新进度，保证刷新后可恢复。

八股关键词：

- HTTP 长连接。
- SSE。
- WebSocket。
- 断线重连。
- 心跳。

面试回答：

> 任务进度是服务端到客户端的单向通知，不需要客户端高频双向通信，所以 SSE 比 WebSocket 更轻。断线后前端用 jobId 重连，后端从 Redis 读最新进度补偿。

### ffmpeg 子进程

业务痛点：

- Java 需要调用 ffmpeg 抽音频。

技术方案：

- ProcessBuilder 启动子进程。
- 消费 stdout/stderr，避免缓冲区阻塞。
- 设置超时和退出码检查。

八股关键词：

- 进程。
- 标准输出阻塞。
- 超时控制。
- CPU 飙高排查。
- 顺序 IO / 随机 IO。

## AI Agent 钩子

### Agent 工具设计

OmniVid Agent 至少包含三个工具：

```text
TranscriptSearchTool
SummaryGenerateTool
VideoCitationTool
```

业务职责：

- `TranscriptSearchTool`：根据问题检索字幕片段。
- `SummaryGenerateTool`：按模板生成总结。
- `VideoCitationTool`：把命中片段转换成可点击引用。

八股关键词：

- RAG。
- Embedding。
- 向量召回。
- 重排。
- Tool Calling。
- Prompt Injection。
- 防幻觉。

面试回答：

> 这个 Agent 不是单纯聊天，而是必须先检索视频字幕，再基于命中片段回答，并返回时间戳引用。如果没有找到足够证据，就提示证据不足。这样能降低幻觉，也能让用户点击引用回到原视频验证。

### 记忆分层

业务痛点：

- 多轮问答需要上下文，但不能无限塞进 prompt。

技术方案：

- Redis 保存短期会话摘要和热点上下文。
- MySQL 保存完整历史消息。
- 向量索引保存字幕语义切片。

八股关键词：

- 短期记忆。
- 长期记忆。
- 上下文窗口。
- Token 成本。
- 语义缓存。

## 简历钩子版本

可以直接放入简历：

- 基于 MySQL 设计长视频解析任务状态机，通过事务、唯一索引和乐观锁保证视频入库、任务创建、状态流转的幂等性与一致性。
- 基于 Redis/Redisson 实现 MD5 防重复提交、分布式锁、任务进度缓存、接口限流和 AI 语义缓存，降低重复解析与重复推理成本。
- 使用 Java 线程池和异步任务模型实现轻量 DAG 解析流水线，并针对任务阻塞、失败重试、线程池拒绝策略进行了工程化处理。
- 针对字幕时间轴检索建立联合索引，并通过 EXPLAIN 优化查询路径，实现视频字幕点击秒级跳转。
- 构建带工具调用能力的 AI Agent，支持字幕检索、总结生成、时间戳引用返回，使回答具备可验证来源。

## 面试官追问兜底句

当被问到还没实现的二阶段能力时，不要硬装已经做完，可以这样说：

> 目前 MVP 版本先用本地线程池 DAG 跑通业务状态机，因为这个阶段更重要的是确定幂等、重试和状态流转边界。RocketMQ 我预留在 dispatcher 层，后续替换为 MQ 消费时，MySQL 状态机和 Redis 进度缓存都可以复用。

> 目前语义缓存可以先用普通 Redis 缓存问题 hash 和答案，向量相似度版本作为增强项。我的设计里已经把缓存 Key、知识库维度和引用结构留好，后续替换成 Redis Vector 或独立向量库不影响业务 API。
